## 1. Какой продукт мы строим

**Forecast** — русскоязычная **prediction market** (прогнозная биржа): пользователь торгует долями исходов **«Да» / «Нет»** по реальным событиям (спорт, крипто, политика и т.д.).

### Референс по UX и механике

[Polymarket](https://polymarket.com) — **образец продукта**, не платформа-зависимость.

### Принципиальные отличия (не обсуждаются в MVP)

| Требование | Смысл |
|------------|--------|
| **Свой финансовый контур** | Балансы, стакан, сделки, выплаты — под нашим контролем. Не маршрутизация ордеров в Polymarket CLOB. |
| **Свои рынки** | События создаём мы; не клон 1:1 их маркетов без юридической оценки. |
| **Аудитория** | Русскоязычные пользователи, в т.ч. РФ — без опоры на их геоблок и API для исполнения. |
| **Этапность** | Сначала доказать продукт на **тестовых деньгах**, затем реальные платежи / on-chain. |

### Что НЕ строим

- Обёртку на `clob.polymarket.com` как основной продукт для РФ.
- «Копию сайта» без своего settlement и правил резолва.
- Букмекерскую модель с фиксированным коэффициентом без вторичной торговли долями (если только не отдельный осознанный продуктовый пивот).

### Текущая стадия (зафиксировано)

**MVP v0** — веб, Supabase, виртуальный USD, 3 seed-рынка, лимитный стакан в PostgreSQL, админ-резолв.

## 2. Продуктовая модель (одним абзацем для агента)

Пользователь видит **рынок** (вопрос + исходы). Цена доли «Да» от 1¢ до 99¢ ≈ **вероятность**. Покупка доли = ставка на исход с возможностью **перепродать** в стакане до резолва. После резолва выигрышная доля платит **$1**, проигрышная — **$0**. В MVP деньги виртуальные; логика та же, что у целевого продукта с реальным контуром.

## 3. Архитектура (целевая vs сейчас)

```text
СЕЙЧАС (MVP v0)                    ЦЕЛЬ (позже)
─────────────────                  ─────────────
Next.js (Vercel)                   Next.js + mobile?
       │                                  │
Supabase Auth                      Auth + KYC
       │                                  │
PostgreSQL:                        PostgreSQL + optional:
  balance, orders,                    on-chain CTF settlement
  trades, positions                 (fork ctf-exchange)
       │                                  │
Admin SQL/UI resolve               Oracle / UMA adapter / council
```

**Правило для разработки:** новые фичи не должны закладывать жёсткую зависимость от Polymarket API для **исполнения** сделок. Gamma/CLOB Polymarket — только **read-only справочник** (идеи, ориентиры цен), если явно не согласовано иное.

## 8. Правила для AI-агента при работе в репозитории

1. Прочитать **§1–§4** этого файла — понять, что Forecast ≠ обёртка Polymarket.  
2. Не предлагать «быстрый старт через их CLOB» для целевой аудитории РФ как основной путь.  
3. Любая торговая логика — через наши RPC/таблицы, пока не начат блок E5.  
4. Переиспользовать OSS по [OPEN_SOURCE.md](./OPEN_SOURCE.md); не дублировать протокол с нуля без причины.  
5. Перед крупной фичей — проверить [RESEARCH.md](./RESEARCH.md) на свежие версии SDK.  
6. Обновлять статусы в §4 при завершении блока.  
7. Пользовательский UI — **русский** по умолчанию.

## 9. Глоссарий

| Термин | Значение |
|--------|----------|
| **Доля / share** | Токен исхода (Да или Нет) |
| **Резолв** | Фиксация победившего исхода |
| **Стакан** | Открытые лимитные заявки |
| **CTF** | Conditional Token Framework (Gnosis) |
| **CLOB** | Central Limit Order Book |
| **Контур** | Вся цепочка: баланс → торговля → выплата |
