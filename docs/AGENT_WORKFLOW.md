# Работа AI-агента: спринты, handoff, новый чат

> Дополняет [AGENTS.md](../AGENTS.md) и [PROJECT.md](./PROJECT.md) §8.  
> Цель: не смешивать в одной сессии auth, админ-RPC, UX и on-chain; после крупного блока — чистый handoff следующему агенту.

---

## 1. Единица работы

| Термин | Смысл |
|--------|--------|
| **Крупный блок** | Подпункт из §4 `PROJECT.md` (D3, B7…) или связанный набор из [ADMIN.md](./ADMIN.md) (например A9+A15 «админ-оператор») |
| **Спринт в чате** | Один продуктовый результат: можно принять, закоммитить (по просьбе пользователя), описать в handoff |
| **Мелкая задача** | Правка после «принято, но…», баг деплоя только что смерженной фичи — **тот же чат** |

**Не нужен** новый агент на каждую мелочь внутри спринта (тест, copy, lint).

---

## 2. Когда новый чат, когда продолжать

### Новый агент (+ бриф в начале сообщения)

- Старт **другого крупного блока** (другая область кода или конфигурации).
- Долгая сессия, агент путает уже сделанное с новой задачей.
- Чистый PR / ревью без «хвостов» прошлых попыток.

### Тот же агент

- Доработка **только что принятого** блока.
- Один PR, одна ветка, одна тема.
- Уточнения по тому же экрану или RPC.

## 3. Рекомендуемая нарезка блоков

| Блок | Чат | Не смешивать с |
|------|-----|----------------|
| **docs/UX.md** + опционально 1 экран-пилот | UX / продукт | D3, миграции, neg-risk |
| **UI по UX** (каталог, рынок, кабинет) | UI | Auth, E5, тяжёлая админка |
| **D3** Magic link / OAuth | Auth | UX-док, neg-risk, админ-RPC |
| **A9 + A15** (дашборд + напоминания резолва) | Админ-оператор | D3, B7; A15 с email — отдельно, если тянет внешние сервисы |
| **A4, A7** и др. мелочи админки | Админ (пакетом) | D3, B7 |
| **B7 neg-risk** | Торговый движок | Всё остальное; сначала [RESEARCH.md](./RESEARCH.md) |

**Порядок по продукту (часто удобнее):** UX-гайд → UI → D3 → админ-backlog → neg-risk / E5.

## 4. Бриф для нового чата (копипаст)

Пользователь или предыдущий агент вставляет в **первое сообщение** нового чата:

```text
Forecast MVP — новый спринт

Репо: github.com/alexiva-85/forecast-mvp, ветка main
Prod: https://forecast-mvp-pied.vercel.app

Обязательно прочитать: docs/PROJECT.md §4, AGENTS.md, docs/AGENT_WORKFLOW.md
[+ docs/ADMIN.md / docs/UX.md — если релевантно]

Блок: [ID и название, напр. D3 — Magic link / OAuth]

Сделать:
- …
- …

Уже сделано (не ломать):
- Кабинет: /portfolio, /portfolio/positions|orders|activity, /profile
- Multi-outcome, sandbox + «В каталог», A18 закрытие торгов, A3 аудит резолва
- …

Не делать:
- Polymarket CLOB для исполнения сделок в РФ
- …

Процесс:
- Миграции: npm run db:migrate (не SQL Editor вручную)
- Коммит и push — только по явной просьбе пользователя

UX (если есть UI):
- docs/UX.md при наличии; иначе: действие первым, иерархия строк, сумма справа по знаку (+/−)
```

## 5. Чеклист завершения крупного блока

Перед handoff агент **обязан** пройти пункты, относящиеся к блоку:

- [ ] Scope закрыт: всё из «Сделать» в задаче сделано или явно вынесено в backlog с причиной
- [ ] `npm test` — релевантные тесты зелёные (или зафиксировано, почему не гонялись)
- [ ] При новых миграциях — `npm run db:migrate` применён / описан шаг для пользователя
- [ ] При новых npm-зависимостях — запись в [OPEN_SOURCE.md](./OPEN_SOURCE.md)
- [ ] При закрытии подпункта §4 — статус в [PROJECT.md](./PROJECT.md) §4
- [ ] При задаче из [ADMIN.md](./ADMIN.md) — строка backlog обновлена (`done` + кратко что в коде)
- [ ] При исследовании внешних API/OSS — `docs/research/YYYY-MM-DD.md` по [RESEARCH.md](./RESEARCH.md)
- [ ] Нет секретов в git; env описан в README / комментарии для пользователя

## 6. Handoff после крупного блока (обязательно для агента)

Когда чеклист §5 пройден, агент **в конце ответа** (отдельным разделом) выдаёт:

### 6.1. Краткий итог

1–3 предложения: что сделано и зачем (для пользователя, не список файлов).

### 6.2. Handoff-бриф для следующего чата

Готовый блок **«Скопируйте в новый агент»** по шаблону §4, с заполненными:

- **Блок:** что считать закрытым
- **Уже сделано:** актуальный список (включая только что смерженное)
- **Следующий логичный блок:** одна рекомендация из §4 / ADMIN + 1–2 альтернативы
- **Не делать в следующем спринте:** границы, чтобы не расползся scope

### 6.3. Рекомендация по агенту

Явная фраза:

> **Рекомендуем начать следующий крупный блок в новом чате** с брифом выше. Текущий чат можно оставить только для мелких правок по только что принятому блоку.

Исключение: пользователь в том же сообщении просит **мелкую доработку** по только что сданному — тогда новый чат не обязателен.

## 7. Open source и дизайн

- [OPEN_SOURCE.md](./OPEN_SOURCE.md) — **не** чеклист «всё внедрить»; для MVP v0 релевантны в основном Gamma (read-only) и Sentry.
- **Готового UI/design kit в OSS-реестре нет.** Удобство для пользователя — [docs/UX.md](./UX.md) (когда появится) + приёмка по сценариям, не копирование репозиториев Polymarket UI.
